## 失败恢复与降级策略

当 Arthas 诊断失败时，不要直接连续盲试更复杂命令。优先判断失败属于哪一类。

### 1. 目标对象不明确

#### 现象

- 不知道该选哪个 Agent
- 类名、方法名不确定
- 多个实例可能都相关

#### 正确做法

- 先调用 `list_agents`
- 再调用 `agent_info`
- 类不确定时优先 `arthas_sc`
- 如当前没有专用方法发现工具，不要凭空猜一串方法名

### 2. Arthas 不可用

#### 现象

- 命令执行前无法确认是否已 attach
- 调用结果提示未连接或不可执行

#### 正确做法

- 先调 `arthas_status`
- 如为 `not_attached`，调用 `arthas_attach`
- attach 后再重新进入诊断流程

### 3. 命令无结果

#### 常见原因

- 条件过严
- 目标方法根本没有被触发
- 同步执行等待时间不足
- 类名/方法名匹配错误

#### 正确做法

- 放宽 `condition`
- 检查是否真有流量命中目标方法
- 对 `trace` / `watch` / `stack` 转用异步 session
- 必要时先用 `arthas_sc` 缩小类范围

### 4. 输出过大或噪声过多

#### 常见原因

- 目标模式过宽
- `watch` 表达式过大
- 没有限制观察范围

#### 正确做法

- 收窄类名与方法名
- 缩小表达式，仅看关键字段
- 加入合理过滤条件
- 不要一次性在多个对象层级上展开大对象

### 5. 长命令阻塞或难以收尾

#### 现象

- 一直等待目标调用
- 返回还未结束
- 需要人工停止观察

#### 正确做法

- 使用 `arthas_session_open`
- 用 `arthas_session_exec` 启动命令
- 用 `arthas_session_pull` 轮询增量结果
- 需要停止时调用 `arthas_session_interrupt`
- 结束后调用 `arthas_session_close`

### 6. 当前命令没有专用 MCP tool

#### 正确做法

- 先确认当前仓库确实没有专用工具
- 再使用 `arthas_exec`
- 命令尽量小而确定，避免把复杂、长期运行命令直接塞进同步执行

### 一条总规则

每次失败后，优先切换到**更确定的事实收集动作**，而不是切换到**更复杂的猜测动作**。

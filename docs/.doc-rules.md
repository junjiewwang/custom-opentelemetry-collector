# 📋 docs 文档管理规则

> 本文件定义了 `docs/` 目录下文档的组织规范，所有新增文档**必须**遵循此规则。

---

## 1. 目录结构规范

### 1.1 按日期归类

所有文档**必须**放入以创建日期命名的子目录中，格式为 `YYYY-MM-DD`。

```
docs/
├── .doc-rules.md          ← 本规则文件（固定在 docs 根目录）
├── 2026-04-29/
│   └── my-new-feature.md  ← 示例：2026年4月29日创建的文档
├── 2026-04-30/
│   └── another-doc.md
└── ...
```

### 1.2 日期确定规则

- **日期取文档首次创建的日期**（即当天日期），格式 `YYYY-MM-DD`
- 如果当天目录不存在，先创建目录再放入文档
- 文档后续修改**不需要**移动到新的日期目录，始终保留在首次创建的日期目录中

### 1.3 禁止事项

- ❌ **禁止**将 `.md` 文件直接放在 `docs/` 根目录下（本规则文件除外）
- ❌ **禁止**在日期目录下再创建子目录（保持扁平结构）
- ❌ **禁止**使用非 `YYYY-MM-DD` 格式的目录名

---

## 2. 文件命名规范

### 2.1 命名格式

- 使用**小写英文** + **短横线（kebab-case）** 连接
- 名称应简洁且能反映文档主题
- 文件扩展名统一使用 `.md`

```
✅ cascade-query-optimization.md
✅ fix-double-sidebar.md
✅ mcp-extension-design.md

❌ CascadeQueryOptimization.md    （不要用驼峰）
❌ fix double sidebar.md           （不要用空格）
❌ 级联查询优化.md                  （不要用中文文件名）
```

### 2.2 常用前缀建议

| 前缀 | 用途 | 示例 |
|------|------|------|
| `fix-` | Bug 修复记录 | `fix-double-sidebar.md` |
| `feature-` | 新功能设计/实现 | `feature-user-auth.md` |
| `refactor-` | 重构记录 | `refactor-state-machine.md` |
| `design-` | 架构/方案设计 | `design-service-entity.md` |
| `optimize-` | 性能/体验优化 | `optimize-cascade-query.md` |
| `phase{N}-` | 分阶段实施文档 | `phase0-protocol-freeze.md` |

## 3. 文档内容规范

### 3.1 推荐结构

每篇文档建议包含以下章节（可根据实际情况裁剪）：

```markdown
## 标题

### 背景
（为什么要做这件事）

### 需求 / 目标
（要达成什么效果）

### 方案设计
（怎么做，包含 mermaid 图）

### 实施进展
- [x] 已完成的步骤
- [ ] 未完成的步骤

### 遗留问题
（已知但暂未处理的问题）
```

### 3.2 图表要求

- 文档中涉及流程、架构、关系等内容时，**建议使用 mermaid 图**
- mermaid 图直接嵌入 markdown，不要使用外部图片链接

## 5. 快速操作指南

### 新增一篇文档

```bash
# 1. 获取今天的日期
TODAY=$(date +%Y-%m-%d)

# 2. 创建日期目录（如果不存在）
mkdir -p docs/$TODAY

# 3. 创建文档
touch docs/$TODAY/my-new-doc.md
```

### 查看最新文档

```bash
# 按日期倒序列出所有日期目录
ls -r docs/ | grep -E '^\d{4}-\d{2}-\d{2}$'

# 查看最新日期目录下的文档
ls docs/$(ls -r docs/ | grep -E '^\d{4}-\d{2}-\d{2}$' | head -1)
```
